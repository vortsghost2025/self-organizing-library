Project context (zero‑knowledge view)  
The repository implements a three‑lane, organism‑level continuity system that spans:
Lane	Role
Archivist‑Agent	Governance root. Owns the single‑entry‑point (BOOTSTRAP.md) that defines all structural rules (Structure > Identity, mandatory correction, no‑duplicate logic). It creates and stores the continuity proof that other lanes must trust.
SwarmMind	Execution lane. Runs the actual AI workload. Must verify that it is a faithful reconstruction of a previously anchored session before it can claim governance authority.
Self‑organizing‑Library	Memory lane. Holds the persisted artifacts (documents, indexes, generated assets) that constitute the “phenotype” of the system.
The continuity proof consists of:
1. Constitutional fingerprint – a SHA‑256 hash of the four core governance files (BOOTSTRAP.md, COVENANT.md, GOVERNANCE.md, CPS_ENFORCEMENT.md). This guarantees the structural rule set has not been altered.  
2. Continuity fingerprint – a SHA‑256 hash of a JSON phenotype registry (PHENOTYPE_REGISTRY.json) that lists every library artifact together with its own hash. This guarantees the content of the memory lane is unchanged.  
3. Lineage metadata – parent_session_id, origin_handoff, and a generation timestamp that link the current session back to the previous one, preventing “orphaned” reconstructions.
What was being built
- PHENOTYPE_REGISTRY.json – a JSON witness that enumerates library artifacts and stores their hashes.  
- RUNTIME_STATE.json – a runtime contract stored in the Archivist‑Agent lane that now contains a continuity_proof object (both fingerprints) and a lineage object (place‑holders for parent session info).  
- verify_recovery.sh – a read‑only Bash script placed in the SwarmMind lane that, when executed, performs three checks:  
  1. Confirms the two fingerprints in RUNTIME_STATE.json match those recorded in the latest hand‑off document and recomputed from the registry.  
  2. Verifies the lineage fields are present (non‑null).  
  3. Returns a distinct exit code (0 = SAME_PHENOTYPE, 2 = RECONSTRUCTED_UNTRUSTED, 4 = ABORT).  
The intent is for SwarmMind’s governance resolver (scripts/resolve-governance-v2.js) to invoke verify_recovery.sh before any other governance logic. If verification fails, the resolver aborts, preventing the lane from proceeding with stale or tampered state.
